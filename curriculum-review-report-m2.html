L02-04 Anthropic Tool Use 代码硬编码索引 · P0
P0 L02-04 · 第 136-137 行 response.content[1] 硬编码索引，实际响应不保证 tool_use 在 index 1

当前代码：

tool_result = response.content[1]  # tool_use block

Anthropic 的响应 content 是一个 block 列表，可能包含 TextBlock 和 ToolUseBlock。当使用 tool_choice={"type": "tool", "name": "extract_fruits"} 强制调用时，模型通常（但不保证）先输出一段思考文本再输出 tool_use。硬编码 [1] 在某些情况下会取到错误的 block。

修复建议：按类型搜索 ToolUseBlock： 
tool_result = next(
    block for block in response.content
    if block.type == "tool_use"
)
fruits_data = tool_result.input  # 已是 dict，无需 json.loads

🟡 P1：建议修复（工程缺陷）

P1 L02-01 · 第 97 行 CRISPE 框架 "E (Experiment)" 释义不准确

当前将 E 解释为"如果一次回答不够，请追问更多细节"。CRISPE 框架中 E 的标准含义是让模型提供多种实验性方案/变体供选择，而非追问。当前释义更接近"迭代"而非"实验"。

修复建议：改为"请提供 2-3 种不同思路的方案，标注各自优劣"或"如果有多种可行方案，请列出供选择"。
P1 L02-04 · 第 73-91 行 OpenAI Structured Outputs 示例缺少 additionalProperties: false

OpenAI 的 strict: true 模式要求 Schema 中所有 object 都设置 "additionalProperties": false，否则 API 会报错。当前示例的 schema 缺少此字段，读者直接运行会遇到 400 Invalid schema 错误。

修复建议：在每个 "type": "object" 的 properties 同级添加 "additionalProperties": false。
P1 P2 · Step 7 · 第 234 行 run_version 函数 hasattr 检查有作用域 bug

当前代码：

result_entities = [e.value for e in result.entities] if hasattr(e, 'value') else [e["value"] for e in result.get("entities", [])]

三元表达式条件 hasattr(e, 'value') 中的 e 在此时尚未定义——它在列表推导式内部才存在。这段代码会抛 NameError。

修复建议：根据函数签名区分 v1（返回 dict）和 v2（返回 Pydantic 模型）： 
if hasattr(result, 'entities'):  # Pydantic 模型 (v2)
    result_entities = [e.value for e in result.entities]
else:  # dict (v1)
    result_entities = [e["value"] for e in result.get("entities", [])]
P1 P2 · Step 6 · 第 199 行 空文档测试用例被直接跳过，未真正测试边界情况

test_document_analysis 中 if not tc["input"].strip(): return 直接跳过空输入测试。但 tc_edge_001 存在的意义就是验证系统对空输入的处理——应该测试 analyze_document("") 不崩溃、返回空实体列表，而非跳过。

修复建议：改为： 
if not tc["input"].strip():
    result = analyze_document(tc["input"])
    assert len(result.entities) == 0
    assert len(result.key_points) == 0
    return

🔵 P2：锦上添花（覆盖度改进）

P2 L02-01 缺少"常见反模式"小节
课程展示了好 Prompt 和差 Prompt 的对比，但没有系统列出常见反模式（如：指令模糊、示例格式不一致、输出格式嵌套过深、Prompt 过长导致注意力稀释等）。一个"不该怎么写"的检查清单能帮助读者避坑。
P2 L02-01 缺少 System Prompt vs User Prompt 的分工指引
课程提到"安全规则放 System Prompt"，但没有系统讲解哪些内容应该放 System、哪些放 User、哪些放 Few-shot。M3-L03-01 会深入讲 System Prompt 架构，但 M2 作为入门应该给出基本分工原则。
P2 L02-02 动态示例选择缺少完整可运行示例
select_examples 函数用了 embed() 和 cosine_similarity() 伪函数。建议用 openai.embeddings.create + numpy.dot 给出可运行的完整代码，或明确标注"伪代码，M4 会实现"。
P2 L02-02 缺少"Few-shot 负迁移"的讨论
当示例与目标任务不匹配时，Few-shot 可能比 Zero-shot 更差（负迁移）。课程讲了示例偏差，但没有明确提到"Few-shot 不是万能的"，建议补充一句关于负迁移的提醒。
P2 L02-03 缺少 Auto-CoT 简介
Auto-CoT（Zhang et al. 2022）自动构建 CoT 示例，是 CoT 领域的重要进展。ToT 已经提到了，Auto-CoT 也值得一句话简介，保持前沿覆盖的完整性。
P2 L02-04 缺少 Gemini 的结构化输出支持
课程覆盖了 OpenAI 和 Anthropic 的结构化输出方案，但缺少 Google Gemini 的 response_schema 支持。三家主流 API 都覆盖会更完整。
P2 L02-05 Prompt 文件使用了虚构模型名 "claude-sonnet-5"
第 222 行 "model": "claude-sonnet-5" 是不存在的模型名。应使用真实模型名如 claude-sonnet-4-20250514 或标注为示例占位符。另外缺少"模型版本升级时的回归测试"讨论——换模型后需要重跑测试集验证。

✅ 做得好的地方

维度评价

深入浅出类比生动且准确："开卷考试"、"不写过程直接交卷"、"带参数的函数"。无术语堆砌。

对比表格每节课都有高质量对比表（Few-shot vs 微调、四种结构化输出方法、评估指标分类），信息密度高。
交叉引用前后引用到位：M03-05（缓存）、M16-03（微调）、M5（ReAct）、M13（护栏）、P3（后续项目），形成知识网。
