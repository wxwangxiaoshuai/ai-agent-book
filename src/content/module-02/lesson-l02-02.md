## Few-shot 与 In-Context Learning

In-Context Learning（ICL）是 LLM 最令人惊叹的能力之一——**不需要梯度更新，仅通过在 Prompt 中放入示例，就能改变模型的行为**。这彻底改变了"适配模型"的方式。

### Zero-shot / One-shot / Few-shot

```
Zero-shot:  只给指令，不给示例
One-shot:   给 1 个示例
Few-shot:   给 2-N 个示例
```

**Zero-shot 示例**：
```
将以下文本分类为"正面"、"负面"或"中性"。

文本：这个产品的功能很强大，但价格太贵了。
分类：
```

**Few-shot 示例**：
```
将以下文本分类为"正面"、"负面"或"中性"。

文本：我非常喜欢这个产品，用起来很顺手。
分类：正面

文本：客服态度太差了，等了半小时没人理。
分类：负面

文本：产品功能还行，颜色和图片有点差异。
分类：中性

文本：这个产品的功能很强大，但价格太贵了。
分类：
```

有了前面的示例，模型更容易理解"混合情感 → 中性"的分类逻辑。

### 为什么 ICL 有效

ICL 的机制可以这样理解：

1. **模式匹配**：模型在训练中见过大量"输入-输出对"，示例激活了相关的模式识别能力
2. **注意力引导**：示例让模型的注意力集中在"任务的格式和结构"上，而非"内容本身"
3. **分布外推**：示例帮助模型从训练分布外推到当前任务

> ICL 不是"模型学会了新知识"，而是"模型从示例中识别出了任务模式"。

### 示例的选择策略

**策略 1：覆盖多样性**

示例应该覆盖不同的输入类型和边界情况：

```
# 分类任务：覆盖正面、负面、中性、混合情感
示例 1：纯正面 → 正面
示例 2：纯负面 → 负面
示例 3：中性客观 → 中性
示例 4：正负混合 → 中性
```

**策略 2：难度递进**

从简单到复杂排列示例：

```
# 代码审查任务
示例 1：变量命名问题（简单）
示例 2：逻辑错误（中等）
示例 3：安全漏洞（复杂）
```

**策略 3：格式一致性**

所有示例必须使用完全相同的格式。不一致的格式比没有示例更糟糕。

```
# 好：格式一致
Q: 北京是哪个国家的首都？
A: 中国

Q: 东京是哪个国家的首都？
A: 日本

# 差：格式不一致
Q: 北京是哪个国家的首都？
A: 中国的首都。

问题：东京是哪个国家的首都？
回答：日本的首都。
```

### 示例数量与质量

| 示例数 | 效果 | 适用场景 |
|--------|------|----------|
| 0 (Zero-shot) | 适用于简单任务 | 翻译、摘要、情感分析 |
| 1-2 | 适用于中等复杂度 | 格式转换、分类 |
| 3-5 | 适用于复杂任务 | 代码生成、结构化抽取 |
| 6+ | 边际收益递减 | 极复杂推理、多步逻辑 |

**一个关键发现**：示例的质量远比数量重要。3 个精心挑选的示例，效果好于 10 个随便选的示例。

### 示例偏差与规避

示例会引入偏差——模型可能"过度模仿"示例的模式。

**常见偏差**：
- **标签偏差**：示例中正面评价过多，模型倾向于将所有输入分类为正面
- **长度偏差**：示例回答都很长，模型倾向于生成冗长的回答
- **风格偏差**：示例都是正式语气，模型在需要口语化时也会用正式语气

**规避方法**：
- 示例中各类别的数量均衡
- 在指令中明确说明"示例仅用于展示格式，不要模仿示例的内容风格"
- 用输出格式约束（如 JSON Schema）来限制模型的自由发挥空间

### 动态示例选择（Example Selection）

在生产环境中，不应使用固定的示例集。更好的做法是根据输入动态选择最相关的示例：

```python
def select_examples(query: str, pool: list[dict], k: int = 3) -> list[dict]:
    """从示例池中选择与 query 最相似的 k 个示例"""
    query_embedding = embed(query)
    scored = []
    for ex in pool:
        ex_embedding = embed(ex["input"])
        similarity = cosine_similarity(query_embedding, ex_embedding)
        scored.append((similarity, ex))
    scored.sort(reverse=True)
    return [ex for _, ex in scored[:k]]
```

这种"动态 Few-shot"策略在 RAG 系统中非常常见，可以显著提升检索后生成的质量。

### 要点总结

- ICL 让模型无需微调就能适配新任务，核心是模式识别
- 示例的质量 > 数量，3 个精选示例通常足够
- 示例必须格式一致，不一致的格式比没有示例更差
- 注意示例偏差——均衡各类别、用输出格式约束规避
- 生产环境用动态示例选择替代固定示例集
