## Few-shot 与 In-Context Learning

In-Context Learning（ICL）是 LLM 最令人惊叹的能力之一——**不需要梯度更新，仅通过在 Prompt 中放入示例，就能改变模型的行为**。这彻底改变了"适配模型"的方式。

### Zero-shot / One-shot / Few-shot

```
Zero-shot:  只给指令，不给示例
One-shot:   给 1 个示例
Few-shot:   给 2-N 个示例
```

**Zero-shot 示例**：
```
将以下文本分类为"正面"、"负面"或"中性"。

文本：这个产品的功能很强大，但价格太贵了。
分类：
```

**Few-shot 示例**：
```
将以下文本分类为"正面"、"负面"或"中性"。

文本：我非常喜欢这个产品，用起来很顺手。
分类：正面

文本：客服态度太差了，等了半小时没人理。
分类：负面

文本：产品功能还行，颜色和图片有点差异。
分类：中性

文本：这个产品的功能很强大，但价格太贵了。
分类：
```

有了前面的示例，模型更容易理解"混合情感 → 中性"的分类逻辑。

### 为什么 ICL 有效

ICL 的机制可以这样理解：

1. **模式匹配**：模型在训练中见过大量"输入-输出对"，示例激活了相关的模式识别能力
2. **注意力引导**：示例让模型的注意力集中在"任务的格式和结构"上，而非"内容本身"
3. **分布外推**：示例帮助模型从训练分布外推到当前任务

> ICL 不是"模型学会了新知识"，而是"模型从示例中识别出了任务模式"。

### 示例的选择策略

**策略 1：覆盖多样性**

示例应该覆盖不同的输入类型和边界情况：

```
# 分类任务：覆盖正面、负面、中性、混合情感
示例 1：纯正面 → 正面
示例 2：纯负面 → 负面
示例 3：中性客观 → 中性
示例 4：正负混合 → 中性
```

**策略 2：难度递进**

从简单到复杂排列示例：

```
# 代码审查任务
示例 1：变量命名问题（简单）
示例 2：逻辑错误（中等）
示例 3：安全漏洞（复杂）
```

**策略 3：格式一致性**

所有示例必须使用完全相同的格式。不一致的格式比没有示例更糟糕。

```
# 好：格式一致
Q: 北京是哪个国家的首都？
A: 中国

Q: 东京是哪个国家的首都？
A: 日本

# 差：格式不一致
Q: 北京是哪个国家的首都？
A: 中国的首都。

问题：东京是哪个国家的首都？
回答：日本的首都。
```

### 示例数量与质量

| 示例数 | 效果 | 适用场景 |
|--------|------|----------|
| 0 (Zero-shot) | 适用于简单任务 | 翻译、摘要、情感分析 |
| 1-2 | 适用于中等复杂度 | 格式转换、分类 |
| 3-5 | 适用于复杂任务 | 代码生成、结构化抽取 |
| 6+ | 边际收益递减 | 极复杂推理、多步逻辑 |

**一个关键发现**：示例的质量远比数量重要。3 个精心挑选的示例，效果好于 10 个随便选的示例。

### 示例排序的影响

研究表明，**示例的排列顺序会影响输出质量**——模型对最后一个示例的"记忆"最强（近因效应）。

```
# 推荐排列方式：把最接近真实输入类型的示例放在最后

示例 1：简单输入 → 输出 A（最不相似）
示例 2：中等输入 → 输出 B
示例 3：复杂输入 → 输出 C（最接近真实输入）
---
真实输入 → 输出（受示例 3 影响最大）
```

**工程建议**：如果用动态示例选择（见下文），把检索到的最相似示例放在末尾位置。

### Token 成本考量

每个示例都消耗 token——这直接影响成本和上下文窗口预算。

```
假设每个示例平均 100 token：
- 0 个示例（Zero-shot）：0 额外 token
- 3 个示例（Few-shot）：~300 额外 token
- 10 个示例：~1000 额外 token

按每天 1000 次调用计算：
- 3 个示例 → 多消耗 300K token/天 ≈ 额外 $0.9-4.5/天（视模型而定）
```

**权衡**：简单任务用 Zero-shot 省钱；复杂任务用 Few-shot 提升质量但增加成本。在 Agent 开发中，System Prompt 中的 Few-shot 示例可以被 Prompt Caching 缓存，大幅降低重复成本（详见 M03-05）。

### Few-shot vs 微调：何时用哪个

| 维度 | Few-shot (ICL) | Fine-tuning |
|------|----------------|-------------|
| 适用场景 | 格式约束、风格引导、简单模式 | 领域知识注入、复杂行为塑造 |
| 成本 | 每次调用多消耗 token | 一次性训练成本，推理不增加 token |
| 迭代速度 | 改 Prompt 立即生效 | 需要重新训练 |
| 灵活性 | 随时修改示例 | 模型"固化"了行为 |
| 能力上限 | 引导已有能力 | 注入新能力 |
| 推荐优先级 | **先试 Few-shot** | Few-shot 不够时再考虑 |

> **黄金法则**：永远先试 Few-shot。只有在 Few-shot 无法解决（如需要注入大量领域知识、或推理成本过高）时，才考虑微调。M16-03 会详细讨论微调的工程决策。

### 示例偏差与规避

示例会引入偏差——模型可能"过度模仿"示例的模式。

**常见偏差**：
- **标签偏差**：示例中正面评价过多，模型倾向于将所有输入分类为正面
- **长度偏差**：示例回答都很长，模型倾向于生成冗长的回答
- **风格偏差**：示例都是正式语气，模型在需要口语化时也会用正式语气

**规避方法**：
- 示例中各类别的数量均衡
- 在指令中明确说明"示例仅用于展示格式，不要模仿示例的内容风格"
- 用输出格式约束（如 JSON Schema）来限制模型的自由发挥空间

### 动态示例选择（Example Selection）

在生产环境中，不应使用固定的示例集。更好的做法是根据输入动态选择最相关的示例：

```python
import numpy as np
from openai import OpenAI

client = OpenAI()

def get_embedding(text: str) -> list[float]:
    """调用 OpenAI Embedding API 获取向量"""
    resp = client.embeddings.create(
        model="text-embedding-3-small",
        input=text,
    )
    return resp.data[0].embedding

def cosine_similarity(a: list[float], b: list[float]) -> float:
    """计算余弦相似度"""
    a_arr, b_arr = np.array(a), np.array(b)
    return float(np.dot(a_arr, b_arr) / (np.linalg.norm(a_arr) * np.linalg.norm(b_arr)))

def select_examples(query: str, pool: list[dict], k: int = 3) -> list[dict]:
    """从示例池中选择与 query 最相似的 k 个示例"""
    query_embedding = get_embedding(query)
    scored = []
    for ex in pool:
        ex_embedding = get_embedding(ex["input"])
        similarity = cosine_similarity(query_embedding, ex_embedding)
        scored.append((similarity, ex))
    scored.sort(reverse=True)
    return [ex for _, ex in scored[:k]]

# 使用示例
example_pool = [
    {"input": "这个产品太棒了", "output": "正面"},
    {"input": "质量很差，退货了", "output": "负面"},
    {"input": "功能一般，价格还行", "output": "中性"},
    {"input": "客服态度极差", "output": "负面"},
]

selected = select_examples("包装不错但物流太慢", example_pool, k=2)
# 会选出语义最接近的 2 个示例作为 Few-shot
```

这种"动态 Few-shot"策略在 RAG 系统中非常常见，可以显著提升检索后生成的质量。M4 会深入讲解 Embedding 和向量检索的原理。

> **注意：Few-shot 不是万能的**。当示例与目标任务不匹配时，Few-shot 可能比 Zero-shot 更差——这叫**负迁移**。如果发现加了 Few-shot 后效果反而下降，先检查示例是否与任务类型一致。对于简单任务（翻译、摘要），Zero-shot 往往已经足够。

### 要点总结

- ICL 让模型无需微调就能适配新任务，核心是模式识别
- 示例的质量 > 数量，3 个精选示例通常足够
- 示例必须格式一致，不一致的格式比没有示例更差
- 注意示例偏差——均衡各类别、用输出格式约束规避
- 生产环境用动态示例选择替代固定示例集
- Few-shot 可能负迁移：示例与任务不匹配时，Zero-shot 可能更好
