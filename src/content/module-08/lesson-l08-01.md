## Agent 记忆的分类学

M3 讲过上下文工程——System Prompt、Context 组装、Token 预算。但那里处理的都是**"当前这一次对话"**的内容。真正的 Agent 要跨会话工作：今天告诉你"我对花生过敏"，下周你问"给我推荐晚餐"，它得记得这件事。**没有记忆的 Agent 是金鱼——每 7 秒重置一次。** 本章先把"记忆"这件事的地图画清楚。

### 从人类的记忆说起

人脑不是一块硬盘，而是多套独立的记忆系统协同工作。认知科学把人类记忆分成几类，理解它们能帮你设计 Agent 的记忆架构：

```
┌──────────────── 人类记忆 ────────────────┐
│                                           │
│  短时记忆 (Working Memory)                │
│  ├─ 正在思考的信息（容量 ~7 项）          │
│  └─ 几秒到几分钟，用完即弃               │
│                                           │
│  长时记忆 (Long-term Memory)             │
│  ├─ 情景记忆 (Episodic)：我经历过的事     │
│  │   "上周二我和产品经理吵了一架"          │
│  ├─ 语义记忆 (Semantic)：我知道的事实     │
│  │   "Python 是动态类型语言"               │
│  └─ 程序记忆 (Procedural)：我会做的事     │
│      "怎么骑自行车 / 怎么查 bug"           │
│                                           │
└───────────────────────────────────────────┘
```

这三类长时记忆分工不同：**情景**是"时间线上的经历"，**语义**是"抽离了时间的知识"，**程序**是"怎么做某件事的步骤"。Agent 要真正"记住"，也得有这三套。

### Agent 记忆的五分类映射

把人类记忆模型映射到工程实现：

| 记忆类型 | 人类类比 | Agent 中的实现 | 典型存储 |
|----------|----------|----------------|----------|
| **短期记忆** | 工作记忆 | 当前对话的 messages 列表 | 内存 / 上下文窗口 |
| **长期-情景** | 昨天发生了什么 | 历史对话、任务轨迹 | 向量库 + 时间戳 |
| **长期-语义** | 知道巴黎是法国首都 | 结构化事实、用户画像 | KV 存储 / 知识图谱 |
| **程序记忆** | 会骑车 | 可复用的技能、操作流程 | 技能库 / 工具配方 |
| **工作记忆** | 边想边记的草稿 | 任务中间状态、scratchpad | 上下文 / Checkpoint |

> 关键洞察：**记忆分类不是学术装饰，而是工程需求**。不同类型的记忆有不同的写入频率、检索方式、更新策略和存储成本——混在一起处理是 Agent 记忆混乱的根源。

### 短期记忆：上下文窗口就是工作记忆

短期记忆最简单，就是塞进 prompt 的 `messages`：

```python
messages = [
    {"role": "system", "content": "你是私人助手"},
    {"role": "user", "content": "我对花生过敏"},
    {"role": "assistant", "content": "已记下，后续推荐食物会避开花生"},
    {"role": "user", "content": "推荐晚餐"},
    {"role": "assistant", "content": "推荐麻婆豆腐..."},
]
```

**优点**：模型天然能"看见"全部历史，无需额外检索。
**致命缺陷**：上下文窗口有上限（哪怕 200K 也会满），且**关闭对话就全丢了**。短期记忆解决不了"跨会话记住"。

### 长期记忆：从"对话日志"到"结构化知识"

短期记忆原样存下来不叫长期记忆——那只是更长的对话历史。真正的长期记忆要做**抽取与转化**：把对话里的临时信息，提炼成稳定、可检索的知识。

```
原始对话（短期记忆）：
  用户："我对花生过敏，上次吃了差点进医院"
  → 这是一段对话记录，不是"记忆"

抽取后的记忆（长期-语义）：
  { user_id: u123, fact: "花生过敏", severity: "严重",
    source: "对话", created_at: "2026-07-22" }
  → 这是可检索、可推理的知识
```

**核心区别**：短期记忆是"对话的原文"，长期记忆是"从对话里抽出的结构化事实"。后者才是 Agent 真正"记住"的东西。

### 五类记忆的写入与检索

不同记忆的读写模式差异巨大，决定了存储选型：

| 记忆类型 | 写入时机 | 写入方式 | 检索方式 | 更新频率 |
|----------|----------|----------|----------|----------|
| 短期 | 每轮对话 | 追加 message | 直接全量塞入 | 高（每轮） |
| 情景 | 任务完成时 | 存对话轨迹+时间戳 | 向量相似度+时间过滤 | 中 |
| 语义 | 抽取出新事实时 | 存为 fact 实体 | 向量+精确匹配 | 低 |
| 程序 | 发现可复用流程时 | 存步骤+前置条件 | 意图匹配 | 很低 |
| 工作 | 每步 | 覆盖式写 | 直接读取 | 极高 |

**写入成本 vs 检索成本**是记忆工程的核心权衡：语义记忆写入要做抽取（花 LLM 调用），但检索便宜、推理精准；情景记忆写入便宜（直接存），但检索要向量计算、且容易噪声多。

### 一个最小多记忆架构

把五类记忆组合起来，Agent 的数据流长这样：

```
用户输入
   │
   ├─→ [短期记忆] 当前对话 messages（工作记忆在其中）
   │
   ├─→ [长期-语义] 检索用户事实（花生过敏、偏好）
   │       ↑ 写入：从对话抽取新事实
   │
   ├─→ [长期-情景] 检索相关历史对话
   │       ↑ 写入：存整段对话轨迹
   │
   ├─→ [程序记忆] 检索可复用技能/流程
   │       ↑ 写入：发现可复用步骤时存
   │
   └─→ 组装上下文 → LLM 推理 → 输出
                         │
                         └─→ 触发记忆写入（抽取事实/存轨迹/存技能）
```

**注意**：不是每次对话都要写所有记忆。写语义记忆要判断"有没有新事实"，写情景记忆通常在对话结束时批量存，写程序记忆要在任务完成后反思"这段流程值不值得复用"。无脑写入会让记忆库变成垃圾场（L08-05 详谈遗忘）。

### 选型决策：什么时候需要哪类记忆

不是每个 Agent 都要五类齐全。按场景裁剪：

| 场景 | 必需记忆 | 可省略 |
|------|----------|--------|
| 一次性问答（客服 FAQ） | 短期 | 其余全部 |
| 个人助手（跨会话记住偏好） | 短期 + 语义 | 情景可精简 |
| 研究助手（记住调研历史） | 短期 + 情景 | 语义可弱 |
| 编程 Agent（积累操作经验） | 短期 + 程序 | 语义可弱 |
| 长期陪伴（既要偏好又要历史） | 全部五类 | — |

> 反模式：**一上来就给所有 Agent 都上向量长期记忆**。简单 FAQ bot 用短期记忆 + system prompt 就够了，强行加长期记忆只会增加延迟、成本和调试复杂度。先问"这个 Agent 真的需要跨会话记住东西吗"。

### 与 M3 上下文工程的关系

容易混淆：M3 的"上下文"和 M8 的"记忆"什么关系？

```
上下文工程 (M3)：每次推理时，往 prompt 里放什么
  ├─ System Prompt（静态底座）
  ├─ 当前对话历史（短期记忆）
  └─ 检索注入的资料（RAG/记忆检索结果）

记忆系统 (M8)：跨推理、跨会话，怎么存和取
  ├─ 短期记忆 → 直接进上下文
  └─ 长期记忆 → 检索后才进上下文
```

**记忆是"仓库"，上下文是"工作台"**。记忆系统负责把信息存进仓库并维护它（更新、遗忘、去冲突）；上下文工程负责在每次推理前，从仓库挑出相关的搬到工作台。两者协作，但职责不同。

### 要点总结

- Agent 记忆分五类：短期（对话历史）、长期-情景（经历）、长期-语义（事实）、程序（技能）、工作（草稿）
- 映射自认知科学：不是装饰，是工程需求——不同记忆的写入/检索/更新模式不同
- 短期记忆是"对话原文"，长期记忆是"从对话抽出的结构化知识"——后者才是真正"记住"
- 核心权衡：写入成本 vs 检索成本；语义记忆写入贵但检索精，情景记忆写入便宜但噪声多
- 不是所有 Agent 都要五类齐全——按场景裁剪，别一上来就上向量长期记忆
- 记忆是"仓库"（跨会话存取），上下文是"工作台"（每次推理组装）——两者协作职责不同
