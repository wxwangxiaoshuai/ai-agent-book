## 云端沙箱方案：E2B / Modal / Fly.io Machines

L09-02 用 Docker 自建沙箱够强，但代价是你得运维——镜像管理、预热池、容器 GC、内核漏洞修补、横向扩展。很多团队不想背这个包袱。**云端沙箱即服务**把这一切外包出去：调一个 API，代码就在云端的微 VM 里跑完，你只管收结果。这一节对比主流方案，并用 E2B 实战。

### 自建 Docker vs 云端沙箱：根本权衡

先想清楚为什么要上云端。核心权衡是**控制力 vs 运维负担**：

```
自建 Docker 沙箱：
  ├─ 控制力强：镜像、网络、资源全自己定
  ├─ 数据不出境：敏感数据留在自己机器
  ├─ 无单次调用费用：固定成本
  └─ 运维重：镜像维护、容器GC、预热池、内核补丁、扩容

云端沙箱：
  ├─ 控制力弱：受限于服务商提供的能力
  ├─ 数据出境：代码/数据要发到云端（合规考量）
  ├─ 按量付费：每次调用计费
  └─ 零运维：服务商兜底一切
```

> 决策原则：**数据敏感且量大 → 自建；要快速上线/弹性扩缩 → 云端**。两者不互斥——生产里常见"自建为主、云端兜底高峰"的混合模式。

### 云端沙箱比 Docker 强在哪：VM 级隔离

L09-02 提过 Docker 的软肋：**共享内核**。云端沙箱通常给的是**微 VM（microVM）**——每个沙箱跑在独立轻量虚拟机里，**有自己独立的内核**：

```
Docker 容器隔离：
  ┌─────── 宿主机内核 ───────┐
  │  容器A   容器B   宿主进程  │  ← 共享内核，逃逸风险存在
  └──────────────────────────┘

微 VM 隔离（firecracker/KVM）：
  ┌─── microVM A ───┐  ┌─── microVM B ───┐
  │  独立内核        │  │  独立内核        │  ← 各自独立内核
  │  代码            │  │  代码            │
  └─────────────────┘  └─────────────────┘
         ↑ 由 Hypervisor 强隔离（AWS firecracker 同款技术）
```

**意义**：容器逃逸漏洞在微 VM 里基本失效——就算代码突破了容器层，外面还有一层硬件虚拟化。威胁模型从"半可信代码"升级到"完全不可信代码"也能扛。这就是为什么代码执行平台（如 Replit、用户上传脚本执行）多用微 VM。

### 主流云端沙箱方案对比

| 方案 | 隔离技术 | 启动延迟 | 特点 | 适合 |
|------|---------|---------|------|------|
| **E2B** | firecracker 微 VM | ~150ms | 专为 Agent 代码执行设计，SDK 友好，支持长会话 | Agent 代码执行首选 |
| **Fly.io Machines** | KVM VM | ~秒级 | 通用 VM 平台，可跑任意服务，按需启停 | 需要跑服务而非单段代码 |
| **Modal** | 容器+微VM混合 | 毫秒级（预热） | 函数即服务，偏数据/ML 计算 | 数据管道、自定义运行时 |
| 自建 gVisor | 系统调用拦截（非VM） | 容器级 | Google 的用户态内核，比容器强比VM轻 | 想自建但比 Docker 强 |

> 注意：方案更迭快，选型前务必查最新文档（定价、限制都在变）。这里讲的是**维度和权衡**，不绑定具体厂商当前状态。

### E2B 实战：30 行跑通云端沙箱

E2B 把"在云端微 VM 里跑代码"封装得像本地函数一样简单。装好 SDK（`pip install e2b-code-interpreter`）并配置 API key：

```python
from e2b_code_interpreter import Sandbox

def run_in_cloud(code: str, timeout: int = 30) -> dict:
    """在 E2B 云端微 VM 里执行代码（SDK v2）"""
    # create 后通常约 150ms 级就绪（同区域；以官方文档为准）
    with Sandbox.create(timeout=timeout) as sb:
        execution = sb.run_code(code)
        stdout = "".join(execution.logs.stdout or [])
        stderr = "".join(execution.logs.stderr or [])
        return {
            "stdout": stdout,
            "stderr": stderr,
            "error": execution.error,  # 无错误时为 None
        }

# 用法
r = run_in_cloud("print(sum(range(101)))")
# stdout = "5050\n"
```

**和自建 Docker 的对比**：代码几乎一样长，但背后——你不用管镜像、不用配 cgroup、不用维护预热池、不用补内核漏洞。E2B 把这些都做了，你只付调用费。

### E2B 的进阶能力：长会话与文件

E2B 不只能跑单段代码，它的沙箱可以**保持状态**——一个 Agent 会话内多次调用，前一次的变量、文件还在：

```python
def interactive_session():
    with Sandbox.create(timeout=120) as sb:
        # 先写入数据文件，再读取分析（顺序不能反）
        sb.files.write("/data.csv", b"name,age\nAlice,30\nBob,25")
        sb.run_code("import pandas as pd\ndf = pd.read_csv('/data.csv')")
        # 复用上一步的 df（同一沙箱，状态保留）
        r = sb.run_code("print(df.describe())")
        r2 = sb.run_code("print(pd.read_csv('/data.csv').shape)")
        return "".join(r2.logs.stdout or [])
```

**适用场景**：数据分析 Agent——用户先上传 CSV，再问"算均值"，再问"画图"。每步复用同一沙箱里的 pandas 对象，不用每次重载。这是自建 Docker 用 `docker exec` 才能实现的能力，云端 SDK 直接给你了。

> 注意：长会话沙箱要**显式管理生命周期**——用完 `close()`，否则按运行时间持续计费。生产里配超时自动回收，防泄漏。

### Fly.io Machines：更通用的 VM

Fly.io 不是"代码执行沙箱"而是"通用微 VM 平台"——你可以在它上面跑任何服务。用作沙箱时，它是"起一个 VM 跑你的代码执行镜像"：

```python
# 概念示意（实际用 flyctl 或 Fly API）
# 1. 把你的代码执行镜像推到 Fly
# 2. 按需起一个 Machine 跑它
# 3. 跑完销毁
```

**和 E2B 的区别**：E2B 是"开箱即用的代码执行 API"，Fly 是"你给我 VM，我管跑啥"。Fly 更灵活（能跑整个服务、自定义镜像、自定义网络），但要自己写"如何执行代码"的那层逻辑。

**选型**：要"快速给 Agent 加代码执行能力"→ E2B；要"自定义沙箱运行时、跑复杂服务"→ Fly.io。

### 成本模型：别让沙箱变成账单炸弹

云端沙箱按用量计费，不注意会失控。理解计费维度：

```
计费维度（典型）：
  · 启动次数：每次创建沙箱计费
  · 运行时长：沙箱存在期间按秒/毫秒计费
  · 资源规格：CPU/内存规格越高越贵
  · 流量：数据传入传出

省钱策略：
  1. 复用而非新建：同一会话多次执行复用一个沙箱
  2. 选最小规格：够跑就行，别开大 VM
  3. 立即 close：用完就关，别等超时
  4. 本地兜底：简单计算（print(sum)）本地 Docker 跑，复杂/不可信才上云
```

**反模式**：每轮 Agent 对话都 `Sandbox.create()` 新建一个云端沙箱跑一句 print——账单会很贵。**先判断任务复杂度**，简单的本地跑，需要重依赖/强隔离的才上云。

### 数据出境合规：上云前必须想清楚

把代码和数据发到云端沙箱，意味着**数据离开了你的环境**。合规红线：

| 数据类型 | 上云风险 | 对策 |
|----------|---------|------|
| 用户 PII（身份信息） | 高，可能违反 GDPR/个保法 | 脱敏后再传，或本地执行 |
| 商业机密/代码 | 高，泄露风险 | 用私有部署版，或本地 Docker |
| 公开数据/计算任务 | 低 | 云端无忧 |
| 第三方受控数据 | 看数据协议 | 必须查数据使用条款 |

> 关键认知：**代码本身就是数据**。Agent 写出的代码可能包含从你私有文档里学到的信息——发给云端沙箱等于把这些信息也传出去了。强合规场景，云端沙箱不是选项，必须本地化（自建 Docker 或私有部署的沙箱服务）。

### 选型决策框架

```
你的代码执行需求是？
├─ 低频、长任务、可接受秒级延迟
│   └─ 自建 Docker（L09-02），零调用费、数据不出境
├─ 高频、要低延迟、Agent 需要长会话
│   └─ 云端沙箱（E2B），预热快、托管运维
├─ 完全不可信代码（用户上传任意脚本）
│   └─ 微 VM 级（E2B/Fly/Modal），容器不够
├─ 强合规、数据绝不出境
│   └─ 自建（Docker 或 gVisor 加固），不碰云端
└─ 弹性扩缩、不想运维
    └─ 云端为主 + 自建兜底
```

> 务实路径：**从 E2B 起步验证产品**（最快上线），量大了、成本敏感了、或合规收紧了，再迁回自建 Docker。两者代码抽象一致（都是"run 代码拿结果"），迁移成本可控——这也是为什么 P9 要做"统一 API 封装"，屏蔽底层是 Docker 还是云端。

### 要点总结

- 核心权衡：自建控制力强+数据不出境但运维重；云端零运维但数据出境+按量付费
- 云端沙箱给微 VM 隔离——独立内核，比 Docker 共享内核强，能扛完全不可信代码
- E2B：专为 Agent 代码执行设计，SDK 友好，支持长会话复用——Agent 代码执行首选
- Fly.io：通用微 VM 平台，灵活但要自己写执行层——适合自定义运行时
- 成本模型：按启动次数+时长+规格+流量计费；省钱靠复用、最小规格、立即 close、本地兜底简单计算
- 反模式：每轮对话新建云端沙箱跑 print——按任务复杂度路由，简单本地、复杂/不可信才上云
- 数据出境合规：代码本身也是数据，强合规场景不能上云，必须本地化
- 务实路径：E2B 起步验证产品 → 量大/合规收紧迁回自建 Docker；抽象统一 API 屏蔽底层
